# 📘 wgpu & winit グリッドベース・グラフィックス開発ナレッジドキュメント

このドキュメントは、wgpuの**インスタンシング（Instancing）**およびwinitの入力系を組み合わせた「グリッドベースの2Dグラフィックス（オセロ、倉庫番、ローグライク、タイルマップ等）」を開発するための実践的なナレッジと逆引きリファレンスをまとめたものです。

---

## 🧭 1. 基本コンセプト：なぜインスタンシングなのか？

グリッドベースの描画（例：64マスのオセロ、1000マスのタイルマップなど）を行う場合、各タイルを個別に描画命令（ドローコール）を出してレンダリングすると、CPUとGPUの間の通信ボトルネックにより動作が極端に重くなります。

**インスタンシング**を採用すると、以下のメリットが得られます：
* **ドローコールは常に「1回」**：GPUメモリ上に「1マスの形状（四角形）」と「各マスの位置・色のリスト（インスタンスデータ）」をそれぞれ1つのバッファとして置いておき、1回の命令で一括描画します。
* **CPU側での状態管理が極めてシンプル**：Rust側では単純な `Vec<Cell>` のような1次元の配列や状態構造体（enum）を持つだけでよく、状態が変わったらそのバッファをGPUに一括転送（`write_buffer`）するだけでリアルタイム描画が実現できます。

## 🛠️ 2. 実践・逆引きリファレンス：「○○したいなら、ここをこうしろ！」

### 🔍 ケース①：グリッドのマス目数を変更したいなら（例: 8x8 から 16x16）
#### 👉 ここをこうしろ：
1. `state.rs` の `State::new` や `update_instances` で、ループの上限値と配列サイズを変更します。
2. 座標配置の計算式（幅）を新しいマス目数（例：`16`）に合わせて調整します。

```rust
// state.rs 内の定数を変更
let num_instances_per_row = 16; // 8 から 16 へ
let mut cells = vec![Cell::Empty; 256]; // 16 * 16 = 256 マス

// update_instances 内の配置計算の変更
// (num_instances_per_row - 1) は 15 になるため、割り算の分母を変更
let x_pos = (x as f32 / 15.0) * 1.6 - 0.8;
let y_pos = (y as f32 / 15.0) * 1.6 - 0.8;
```

---

### 🎨 ケース②：マスの大きさや隙間（グリッド線の太さ）を調整したいなら
#### 👉 ここをこうしろ：
1. **マスの絶対サイズ**：`shader.wgsl` の頂点シェーダーでの縮小・拡大率（`model.position * 0.25` の部分）を書き換えます。大きくすると隙間が狭くなり、小さくすると隙間が広がります。
2. **グリッドの全体の広がり**：`state.rs` の `update_instances` 内の配置計算のスケール（例: `1.6` 倍）とオフセット（例: `-0.8`）を調整します。

```wgsl
// shader.wgsl: マス自体の大きさを変える
// 0.25 を大きく（例: 0.28）するとセル同士がくっつき、小さく（例: 0.20）すると隙間が広がります
let scaled_position = model.position * 0.25; 
```
```rust
// state.rs: グリッド全体の表示幅を変える
// 1.6 の数値を小さく（例: 1.2）すると、画面中央にグリッドがギュッと凝縮されます
let x_pos = (x as f32 / 7.0) * 1.6 - 0.8;
```

### 🚀 ケース③：マスの種類（enum）を増やしたいなら（例：壁、火、水、プレイヤーなど）
#### 👉 ここをこうしろ：
1. `Cell` enum に新しい列挙子を追加します。
2. `update_instances` の `match cell` にその新しい列挙子を追加し、対応するカラー（RGB）を定義します。

```rust
// 1. 列挙子の追加
pub enum Cell {
    Empty,
    Black,
    White,
    Wall,  // 新しい壁タイル
    Water, // 新しい水タイル
}

// 2. カラーの定義
let color = match cell {
    Cell::Empty => [0.1, 0.5, 0.1],   // 緑
    Cell::Black => [0.05, 0.05, 0.05], // 黒
    Cell::White => [0.95, 0.95, 0.95], // 白
    Cell::Wall  => [0.4, 0.4, 0.4],   // グレーの壁
    Cell::Water => [0.1, 0.3, 0.8],   // 青い水
};
```

### 🔄 ケース④：各マスを回転させたり、個別に拡大縮小したいなら
#### 👉 ここをこうしろ：
1. インスタンスデータ `InstanceRaw` に、回転角や個別スケール値を保持するフィールドを追加します。
2. GPUに伝えるアトリビュートレイアウト（`InstanceRaw::desc()`）に新しい属性（`shader_location: 4` など）を追加します。
3. `shader.wgsl` でそれらを受け取り、頂点シェーダー側で回転行列やスケールを適用します。

```rust
// state.rs: 構造体の拡張
struct InstanceRaw {
    position: [f32; 3],
    color: [f32; 3],
    rotation: f32, // ラジアン単位の回転角を追加！
}
```
```wgsl
// shader.wgsl: 頂点シェーダーでの回転処理
// location(4) から rotation を受け取る
let c = cos(model.rotation);
let s = sin(model.rotation);
// 回転行列を適用して頂点位置を回転させてから、ワールド座標に配置する
let rotated_x = model.position.x * c - model.position.y * s;
let rotated_y = model.position.x * s + model.position.y * c;
let world_position = vec3<f32>(rotated_x, rotated_y, 0.0) * 0.25 + model.instance_position;
```

### 🗺️ ケース⑤：動的に「テクスチャ（画像）」を貼り分けたいなら
もし単なる単色塗りつぶしではなく、草地や岩などの画像をグリッドごとに貼り分けたい場合：
#### 👉 ここをこうしろ：
1. `InstanceRaw` に `texture_index: u32`（または `uv_offset: [f32; 2]`）を追加し、アトリビュートでシェーダーに送ります。
2. GPU側には、テクスチャを束ねた「テクスチャアレイ（Texture Array）」または大きな「アトラス画像」を用意します。
3. シェーダー内で、インスタンスバッファから送られてきた `texture_index` を用いて、対応するテクスチャをサンプリングします。

```wgsl
// WGSLでのテクスチャアレイのサンプリング例
@group(0) @binding(0) var t_diffuse: texture_2d_array<f32>;
@group(0) @binding(1) var s_diffuse: sampler;

// フラグメントシェーダーで、インスタンスから引き渡されたインデックスを使ってサンプリング
let color = textureSample(t_diffuse, s_diffuse, in.uv, in.texture_index);
```

### ⌨️ ケース⑥：キーボードやクリックのイベントを追加したいなら
#### 👉 ここをこうしろ：
1. `app.rs` の `window_event` 内で、対象の `WindowEvent` をマッチングします。
2. `State` 構造体に状態更新ロジックを実装し、`app.rs` からそれを呼び出します。
3. ロジック実行後、必ず `self.update_instances()` を呼び出して GPU 上のバッファを同期します。

```rust
// 例: Rキーで盤面を全クリアするイベントを追加する場合
// app.rs
WindowEvent::KeyboardInput { event, .. } => {
    if event.physical_key == PhysicalKey::Code(KeyCode::KeyR) && event.state == ElementState::Pressed {
        if let Some(state) = &mut self.state {
            state.clear_board(); // state側で定義したクリア処理
        }
    }
}
```

## 📈 3. 数学の部屋：座標系変換の公式集

グリッドグラフィックスでの最大の難所は「マウスで触った位置」を「ゲームのセル」に変換する計算です。

### 1. ウィンドウ座標 ➔ 正規化デバイス座標 (NDC) の汎用変換式
ウィンドウサイズが $(W, H)$、マウス位置が $(X_{win}, Y_{win})$ のとき：

$$ndc\_x = \frac{X_{win}}{W} \times 2.0 - 1.0$$
$$ndc\_y = 1.0 - \frac{Y_{win}}{H} \times 2.0$$

> [!IMPORTANT]
> ウィンドウ座標系は「左上が $(0,0)$、下方向がYプラス」ですが、グラフィックス座標系（NDC）は「中心が $(0,0)$、上方向がYプラス」です。そのため、Y軸の変換では `1.0 - ...` のように引く順番が反転します。

### 2. グリッド配置と逆引きの一般化公式
グリッドの1辺のセル数を $N$、グリッド全体の表示幅を $S$（画面いっぱいに広げるなら `2.0` など）、中心からのオフセットを $O$ とするとき：

#### セルインデックス $i \in [0, N-1]$ から配置座標 $pos$ を求める式：
$$pos = \frac{i}{N - 1} \times S - \frac{S}{2.0}$$

#### 座標 $pos$ から最も近いセルインデックス $i$ を逆算する式：
$$i = \text{round}\left( \frac{pos + \frac{S}{2.0}}{S} \times (N - 1) \right)$$

## 🔍 4. トラブルシューティング：「こんなときはここを疑え！」

### 🚨 画面が真っ暗で何も表示されない！
* **疑うべき箇所①**：頂点データ（`VERTICES`）の向き（カリング）。wgpuのデフォルト設定では「裏面を向いているポリゴン」は描画から除外（カリング）されます。頂点の並び順が「反時計回り（CCW）」になっているか確認してください。時計回りになっていると、裏面と判定されて画面から消えます。
* **疑うべき箇所②**：`update_instances()` が初期化時に呼ばれているか。バッファを動的書き換えする設計にした場合、起動直後に1回呼び出さないと、バッファがすべてゼロデータ（重なって真っ黒、または範囲外）になり何も表示されません。

### 🚨 特定のセルをクリックしたのに、違うセルが反応する！
* **疑うべき箇所**：ウィンドウサイズ $(W, H)$ が最新のものになっているか。ウィンドウサイズがリサイズされた際、 `State` の `config.width` と `config.height` を更新（`resize()` メソッドを実行）していないと、クリック座標の割り算の分母がズレるため、クリック判定が大きく狂います。

### 🚨 リサイズした瞬間、またはウィンドウを最小化した瞬間にアプリがクラッシュする！
* **疑うべき箇所**：最小化時のサイズゼロチェックが入っているか。winitは最小化された際にサイズ `(0, 0)` をイベントとして発行します。これをそのまま `surface.configure()` に流すと、wgpuはテクスチャサイズ0のエラーで確実にパニックします。必ず `width > 0 && height > 0` のガードを入れてください。
