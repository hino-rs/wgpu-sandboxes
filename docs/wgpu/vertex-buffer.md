# 頂点バッファ

VRAMに確保された、頂点データのバイト列を受け取る箱。

## なぜ必要なのか

シェーダーに座標や色の情報をハードコードすると、動的なモデルの変更や外部からの3Dモデルを読み込めない。

以下のように、CPUで情報を管理し、GPUは描画だけを担当することで、柔軟性を上げることができる。

- CPU側で「頂点データ(位置や色など)」を定義し、それをGPUメモリ(頂点バッファ)に転送する。
- GPUは、描画命令が呼ばれた際に、そのバッファからデータを順番に読み込んでシェーダーに渡す。

## 頂点データをどう定義するか

通常、1つの頂点は**座標・色・テクスチャ座標**など、複数の情報を持つ。
Rust側では、これらを1つの構造体として定義するのが一般的(インターリーブ配置)。

```rust
#[repr(C)]
#[derive(Copy, Clone, Debug, bytemuck::Pod, bytemuck::Zeroable)]
struct Vertex {
    position: [f32; 3], // 位置座標 (x, y, z)
    color: [f32; 3],    // 色 (r, g, b)
}
```

> [!NOTE]
> 1. `#[repr(C)]` は、メモリレイアウトを書いた通りに固定するために使う。
> 2. `bytemuck::Pod`と`Zeroable`は、Rustの構造体を安全に生のバイト列にキャストするためのライブラリ。GPUにデータを送るサイオは、構造体をただのバイトの塊として転送する必要があるため。
> - `Pod`: "Plan Old Data"の略。ポインタなどを含まない、ただの数値の塊であれうことを示す。
> - `Zeroable`: 全てビット0にできることを示す。

## GPUにデータの見え方を教える (レイアウト定義)

GPUに生のバイト列を送りつけるだけでは、「どこからどこまでが位置座標で、どこからが色なのか」を理解できない。
そのため、レンダーパイプラインに対して、「頂点バッファのレイアウト」を登録する必要がある。
wgpuでは、`wgpu::VertexBufferLayout`を使って表現する。

```plaintext
バイト列: [ X1, Y1, Z1, R1, G1, B1 | X2, Y2, Z2, R2, G2, B2 | ... ]
         |<------ 頂点1回分 ------>|<------ 頂点2回分 ----->|
         |<-位置3個-->|<--色3個--->|
```

- ストライド: 頂点1個分のデータの大きさ。GPUが次の頂点に進むために「何倍とスキップすればいいか」を示す(例: `std::mem::size_of::<Vertex>()` = 24バイト)。
- アトリビュート: 頂点データの中に入っている個々の属性の情報。
    - 位置情報 (Location 0): 先頭 (オフセット0)から始まり、f32が3つの形式
    - 色情報 (Location 1)d: `position`の後ろ (オフセット12バイト)から始まり、f32が3つ


## wgpuでの実装5ステップ

1. 頂点構造体とデータの定義
    CPU側(Rust)で頂点データを用意する。
2. 頂点バッファの作成とデータ転送
    `device.create_buffer_init`を使用して、GPU上にメモリ領域を確保し、Rustの頂点データを書き込む。このバッファは`wgpu::BufferUsages::VERTEX`(頂点バッファとして使うよ、という宣言)を指定する。
3. パイプライン記述の変更
    `render_pipeline`を作成する際、`vertex.buffers`に頂点バッファのレイアウトを登録する。
4. シェーダー(WGSL)の変更
    シェーダーの入力引数を、バッファから渡されるアトリビュート(`location(0) position: vec3<f32>`など)に変更する。
5. 描画処理でのバインディング
    描画命令を発行する際、`render_pass.set_vertex_buffer(0, self.vertex_buffer.slice(..))`のようにして、作成したバッファをGPU二セットし、`draw`を呼び出す。
